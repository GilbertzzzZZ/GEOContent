# GEO 文章写作模板（带注释版）

> 直接复制使用，按注释填写。每个元素都注明了"为什么有效"。
> 配合 `geo-content-playbook.md` Part 1 使用。

---

## 模板结构

```markdown
# [文章标题：包含目标关键词，20字以内]

<!-- 
⚠️ 为什么：标题是 AI 检索的第一匹配层。清晰的标题让 AI 准确判断"这篇文章回答了什么问题"。
✅ 好标题："2026年房贷利率计算完全指南"
❌ 坏标题："关于那些你可能不知道的房贷小知识"
-->

**作者：[真实姓名]，[职称]，[机构名]**  
**更新日期：[YYYY-MM-DD]**

<!-- 
⚠️ 为什么：具名作者 + 职称 = E-E-A-T 的 Expertise 信号。AI 用作者资质判断内容可信度。
匿名内容在 AI 引用竞争中处于劣势。（来源：Semrush 2026 指南）
更新日期告诉 AI 爬虫这是最新内容，AI 偏好近期更新的内容。
-->

---

## 引言段（定义性开头，150字以内）

[X] 是 [Y 的定义]。[核心数据点：具体数字]。[本文将覆盖的内容范围]。

<!-- 
⚠️ 为什么：这是整篇文章中最重要的段落。
- AI Overview 文字往往直接提取页面引言段（来源：DSLD Mortgage 案例）
- "X 是 Y" 定义性句式是 AI 回答"什么是X"查询时的首选提取格式
- 开篇就给数据，因为统计数据提升 AI 可见度 30-40%（GEO 原始论文）
- 150字以内，因为 AI 提取的是信息块，不是整页

✅ 示例：
"GEO（Generative Engine Optimization）是针对 AI 搜索引擎优化内容可见性的方法论。
研究表明，包含统计数据的内容在 AI 响应中可见度提升 30-40%。
本文将从写作结构、权威信号、渠道策略三个维度，系统讲解如何让内容被 AI 引用。"

❌ 反面：
"在当今这个数字化转型的时代，随着人工智能技术的飞速发展，我们不得不思考一个问题……"
（AI 不会引用这种废话铺垫）
-->

---

## H2：[第一个核心子话题]

### H3：[子话题的第一个要点]

[一段 100-200 字的解释，包含至少 1 个具体数据]（来源：[权威出处]）。

<!-- 
⚠️ 为什么：
- H2/H3 层级让 AI 按标题分块检索，每个标题 = 一个可独立引用的信息块
- 每段都有数据 + 来源标注 = 可验证 = 可信（引用权威来源是 GEO Tier 1 要素）
- 100-200 字 = 一个完整信息块，不长不短，刚好够 AI 提取
-->

**关键要点：**

- 要点 1：[动词开头，一句话说清]
- 要点 2：[包含具体数字]
- 要点 3：[引用权威来源]

<!-- 
⚠️ 为什么：子弹点列表比散文式描述被 AI 引用率更高。
AI 需要"可直接嵌入响应的信息块"，子弹点天然就是。
-->

> "[专家姓名]，[职称]，指出：'[直接引语]'"

<!-- 
⚠️ 为什么：直接引语是 GEO 的 Tier 2 要素。
AI 可以将引语直接嵌入回答，作为"专家观点"呈现，提升回答的权威感。
-->

### Q1：[用户可能搜索的完整问题]？

[直接回答，50-100字，开头用定义性语句]。[补充数据或例子]。

### Q2：[另一个常见问题]？

[直接回答]。

### Q3：[长尾变体问题]？

[直接回答]。

<!-- 
⚠️ 为什么：FAQ 结构天然契合 AI 的"查询-回答"逻辑。
- 覆盖 3-5 个长尾问题，扩大内容被不同查询匹配的概率
- 每个回答都是独立的"引用就绪"信息块
- 配合 FAQPage Schema 标记，让 AI 系统精准解析
- 问题要用用户的自然语言，不要用行业黑话
-->

## Schema Markup 模板

<!-- 根据内容类型选择，嵌入页面 <head> 或 <script type="application/ld+json"> -->

### Article Schema
```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "[文章标题]",
  "author": {
    "@type": "Person",
    "name": "[作者姓名]",
    "jobTitle": "[职称]",
    "url": "[作者主页URL]"
  },
  "datePublished": "[YYYY-MM-DD]",
  "dateModified": "[YYYY-MM-DD]",
  "publisher": {
    "@type": "Organization",
    "name": "[品牌名]"
  }
}
```

### FAQPage Schema
```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "[问题1]",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "[回答1]"
      }
    }
  ]
}
```

### HowTo Schema
```json
{
  "@context": "https://schema.org",
  "@type": "HowTo",
  "name": "[指南标题]",
  "step": [
    {
      "@type": "HowToStep",
      "name": "[步骤名]",
      "text": "[步骤描述]"
    }
  ]
}
```
```

## 使用说明

1. 复制上方模板，删除所有 `<!-- -->` 注释块
2. 按照 `[]` 占位符填入实际内容
3. 每个 H2 区块控制在 200-400 字
4. 全文建议 1,000-2,000 字（LinkedIn 研究显示此区间被引最多）
5. 完成后用 `geo-optimization-checklist.md` 逐项核对
